Phase 01 — June 1 to June 28
01

The Engine Room

Master the building blocks before you build the house.

June 1 – June 28 Weeks 1–4 80–100 hrs Python · SQL · Git · Data Formats

Every DE who is fast later was patient here. This phase is about depth, not speed. You are not racing through tutorials — you are building muscle memory. By the end of June you should be able to open a Python file and write real code without googling every line.

WEEK 1 · Jun 1–7
Python Deep Dive
File I/O, pandas, list comprehensions, OOP basics, virtual environments. Stop watching. Start writing.
WEEK 2 · Jun 8–14
SQL Mastery
Window functions, CTEs, subqueries, EXPLAIN plans. Write 50 queries on a real dataset. Not toy examples.
WEEK 3 · Jun 15–21
Data Storage & Formats
Relational vs NoSQL. CSV vs JSON vs Parquet vs Avro. Compression. Why each format exists and when to use it.
WEEK 4 · Jun 22–28
Dev Workflow + Projects
Git, GitHub, CLI fluency, project structure, virtual environments, requirements.txt. Build P1-A and P1-B.

Phase 1 Milestones
MILESTONE P1-A · WEEK 4
Data Profiler CLI
Core Build
What You Build A command-line tool: point it at any CSV file or database table and it outputs a full data quality report — null percentages, inferred data types, duplicate rows, value distributions for each column, and a list of suspected quality issues.
Why This Matters This becomes a reusable tool you will use in every single phase that follows. It also proves you can build something useful from scratch, not just follow a tutorial. Data profiling is the first thing any DE does on a new dataset.
Suggested Dataset Download a messy real-world CSV from Kaggle — the Global Terrorism Database, the NYC Taxi dataset, or any government open data CSV with 50k+ rows.
Stack Python (pandas, argparse, tabulate), SQLite or PostgreSQL for the database mode, GitHub repo with a proper README.
Run it on a CSV you have never seen before. It should tell you something useful you didn't already know about the data.
MILESTONE P1-B · WEEK 4
SQL Analytics Engine
Core Build
What You Build Load a real dataset into PostgreSQL. Write 15 analytical SQL queries covering: basic aggregations, multi-table JOINs, window functions (RANK, LAG, LEAD, running totals), CTEs, and at least one complex nested query. Export the results as an HTML report with a clear business question for each query.
Why This Matters SQL is the most-used skill in DE. Window functions and CTEs appear in almost every interview and every real pipeline. You need these to be automatic.
Suggested Dataset StackOverflow Developer Survey (kaggle.com) or a public sales/e-commerce dataset with multiple related tables. You want JOINs to be natural, not forced.
Stack PostgreSQL, pgAdmin or DBeaver (free), Python to generate the HTML report. Push the SQL files to GitHub.
Show someone the report without explaining the SQL. If they can follow the insights, your queries are good.
Phase 02 — June 29 to July 26
02

The Architect

Build systems that run while you sleep.

June 29 – July 26 Weeks 5–8 80–100 hrs Airflow · dbt · Docker · Data Modeling

This is where hobbyists and engineers diverge. A hobbyist writes a script. An engineer builds a system. This phase is about orchestration, modelling, and containerisation — the three things that make pipelines production-grade. Expect to spend a full day just getting Airflow running. That day is worth it.

WEEK 6 · Jul 6–12

WEEK 7 · Jul 13–19
Docker + Compose
Images, containers, volumes, networking. Dockerfile, docker-compose. Containerise your Phase 1 work. This is the unlock.
WEEK 8 · Jul 20–26
Pipeline Assembly
Wire everything together. Airflow + PostgreSQL + dbt + Docker in one system. Build P2-A, P2-B, P2-C.

Phase 2 Milestones
MILESTONE P2-A · WEEK 6
Automated Multi-Source Pipeline
Intermediate
What You Build An Airflow DAG that runs on a daily schedule. It pulls from two different public APIs and one CSV file, merges the data, runs your P1-A Data Profiler on the result, logs the quality report, and writes clean data to PostgreSQL. If Task 2 fails, Task 3 doesn't run.
Why This Matters This is the core skill loop: ingest → validate → store → schedule. Every DE job in the world does some version of this. Airflow is on nearly every job description.
Suggested Sources Source 1: Open-Meteo API (weather, no key). Source 2: CoinGecko API (crypto prices, free). Source 3: World population CSV from Kaggle. Join weather + currency movement as an economics curiosity.
Stack Apache Airflow, PostgreSQL, Python (pandas, requests), pgAdmin.
The DAG has run at least 3 times automatically. You can see execution history in the Airflow UI. You can trigger a backfill.
MILESTONE P2-B · WEEK 7
dbt Transformation Layer
Intermediate
What You Build On top of P2-A's raw data in PostgreSQL, build a proper star schema using dbt: staging models that clean raw tables, intermediate models for joins, and final fact + dimension tables. Add dbt tests (not null, unique, accepted values) and generate the documentation site.
Why This Matters dbt is how modern data teams do SQL transformations in a maintainable, testable, version-controlled way. Knowing dbt separates senior DEs from juniors in 2025+.
Stack dbt Core (free, pip install), PostgreSQL, your P2-A data, GitHub for version control.
Run "dbt docs generate && dbt docs serve" and show someone the lineage graph. Your schema should make sense to a non-technical person.
MILESTONE P2-C · WEEK 8
The Containerised Stack
Intermediate
What You Build Package your entire Phase 2 system — Airflow, PostgreSQL, and dbt — into a single docker-compose.yml file. Anyone should be able to clone your repo, run "docker-compose up", and have your full pipeline running in under 5 minutes.
Why This Matters Docker is the bridge between local development and real deployments. Not knowing Docker is a red flag in any DE interview. This project proves you can ship, not just script.
Stack Docker Desktop, docker-compose, Dockerfile, your Phase 2 code, GitHub.
Delete your local database and containers. Run "docker-compose up" from scratch. Everything works. You get the same results.
Phase 03 — July 27 to August 23
03

Cloud & ML Interface

Learn how data engineers feed AI teams.

July 27 – August 23 Weeks 9–12 80–100 hrs GCP · BigQuery · Feature Engineering · Spark intro

This is the phase most DEs skip, and it's why most DEs are not valuable on AI teams. Understanding how your pipelines connect to model training — what shapes ML engineers need, how feature stores work, what "training-ready data" means — is the specific thing that makes you useful in October.

WEEK 9 · Jul 27–Aug 2
Cloud Fundamentals
GCP or AWS basics. Object storage (S3 / GCS), IAM, cloud CLI. Free tier is enough. Set up once, use everywhere.
WEEK 10 · Aug 3–9
Cloud Data Warehousing
BigQuery or Redshift. Columnar storage, partitioning, clustering, cost awareness. Migrate your Phase 2 pipeline.
WEEK 11 · Aug 10–16
Feature Engineering for ML
What do models actually need from data? Feature stores, train/test splits, label encoding, scaling, Parquet output. This is the ML handoff.
WEEK 12 · Aug 17–23
PySpark + Streaming Intro
PySpark for large-scale transforms. Kafka basics — producer, consumer, topics. Not deep, but fluent enough to read and talk about it.

Phase 3 Milestones
MILESTONE P3-A · WEEK 10
Cloud Data Warehouse
Advanced
What You Build Migrate your Phase 2 pipeline to load data into BigQuery (free tier, 10GB/month included). Build a Looker Studio dashboard (free, connects directly to BigQuery) showing at least 4 meaningful charts. Set up partitioning on the main table by date.
Why This Matters BigQuery is the most commonly used cloud DW in DE roles worldwide. Partitioning + columnar storage = understanding of scale. Looker Studio = you can present results to non-technical stakeholders.
Stack Google Cloud Platform (free tier), BigQuery, Cloud Storage, Looker Studio, Python (google-cloud-bigquery SDK).
Your Airflow DAG now writes to BigQuery instead of local PostgreSQL. The Looker Studio dashboard auto-refreshes when new data arrives.
MILESTONE P3-B · WEEK 11
Feature Engineering Pipeline
Advanced
What You Build Take a Kaggle ML competition dataset (pick one with a clear prediction task). Build a clean Python pipeline that reads the raw data, engineers at least 10 new features (aggregations, ratios, date parts, lag features, encodings), runs data quality checks, and outputs a train.parquet and test.parquet ready to hand to an ML engineer. Write a short README explaining each feature and why you created it.
Why This Matters This is the highest-value thing you can do for an AI competition team. Most teams spend 60% of their time on feature engineering. If you arrive with this pipeline ready to run on any dataset, you are immediately the most useful person on the team.
Suggested Dataset Kaggle: House Prices (regression), Titanic (classification), or any tabular competition. Pick something with mixed types — numbers, categoricals, dates — so all engineering types are represented.
Stack Python (pandas, scikit-learn, pyarrow), Parquet output, GitHub.
An ML engineer can clone your repo, run one script, and get train.parquet and test.parquet. The Parquet files are the only deliverable they need.
MILESTONE P3-C · WEEK 12 · STRETCH
Mini Streaming Pipeline
Stretch Goal
What You Build Set up a local Kafka cluster via Docker. Write a Python producer that sends messages (simulated sensor data or API events), a consumer that reads and writes to PostgreSQL, and a simple dashboard that shows the last 100 events in real time.
Why This Matters for Competitions Some AI competitions involve real-time data. Even if you don't use Kafka in October, knowing the vocabulary and flow means you can talk intelligently with any ML team about streaming vs batch tradeoffs.
Stack Apache Kafka (Docker), Python (kafka-python), PostgreSQL, Streamlit for the live view.
Messages flow from producer to consumer to database. The dashboard shows new records arriving without manual refresh.

Phase 4 Milestones
MILESTONE P4-A · WEEKS 13–14
Kaggle Competition Entry
Competition
What You Build Enter an active Kaggle tabular competition. Build a modular, reusable data pipeline: raw data → EDA notebook → cleaning script → feature engineering → train-ready Parquet. The pipeline should be parameterised so a teammate can swap in a different dataset and re-run it.
The Real Goal Not the leaderboard position. The goal is building a pipeline you could hand to a teammate and they'd understand it in 10 minutes. Document every feature engineering decision. Write a short "data brief" — a 1-page summary of what's in the data, what you found, and what you think matters for the model.
Suggested Competitions Look for "Getting Started" or active tabular competitions on kaggle.com/competitions. Avoid image-only or NLP-heavy competitions — you want tabular data.
Stack Python (polars or pandas, pyarrow), DuckDB for fast SQL on the raw files, GitHub, Jupyter.
Your pipeline processes the full competition dataset in under 60 seconds. Your data brief is readable in under 5 minutes by a non-specialist.
MILESTONE P4-B · WEEKS 15–16
Production Capstone
Capstone
What You Build A complete, production-grade DE system on a topic you find genuinely interesting. It must include: automated ingestion from at least 2 sources, a cloud data warehouse (BigQuery), dbt transformations with tests, an Airflow schedule, data quality monitoring (alert when checks fail), and a public dashboard. The whole thing runs in Docker and is on GitHub with a README that reads like a proper project, not a tutorial log.
Why This Is Your Most Important Asset This is what you show to competition teams, employers, and clients. Not your certificates. Not your DataCamp profile. This project. Make it something you're proud to open on a laptop in front of strangers.
Topic Ideas Nigerian electricity grid data + weather correlation · African startup funding trends · Football performance analytics · Global supply chain indicators · Anything that gives you energy when you think about it.
Stack Python, Airflow, dbt, BigQuery, Docker, Great Expectations or custom data quality checks, Looker Studio or Metabase, GitHub.
You could present this project confidently in a 5-minute lightning talk to people who don't know you. The README sells it before you open your mouth.
MILESTONE P4-C · WEEK 17
48-Hour AI Competition Simulation
Sprint
What You Build On a Saturday morning, pick a dataset you have never seen before. You have until Sunday evening. Build and "present" a complete AI-ready data pipeline: ingest → profile → clean → feature engineer → output training-ready Parquet files → write a 1-page data brief. Act as if you are the DE on a real competition team and your ML teammates are waiting on you.
Why This Matters The October competition will not give you time to think slowly. You need to move fast and make good decisions simultaneously. This simulation is that stress-test. The goal is not a perfect pipeline. The goal is a complete one.
Rules No tutorials, no asking for help, no familiar dataset. Use all the tools you have learned. Commit to GitHub every 4 hours. Write the data brief at the end as if you're handing it to a real team.
You finish. Even if it's messy. You deliver something. The act of finishing under pressure is the skill you're building.

Reading List — All Four Books
PHASES 1–3 · CORE TEXT

Fundamentals of Data Engineering

Joe Reis & Matt Housley · O'Reilly 2022

Read cover to cover, spread across phases. This is your DE bible. Every chapter introduces a concept you will implement in code that week.

PHASE 1 · FOUNDATIONS

Python for Data Analysis

Wes McKinney · O'Reilly · 3rd Edition

Ch 1–10 in Phase 1. The definitive pandas guide. Read with your editor open — type every example yourself.

PHASE 2 · PIPELINES

Data Pipelines Pocket Reference

James Densmore · O'Reilly 2021

Read cover to cover in Phase 2. Short, dense, and directly practical. Zero filler. Everything in it shows up in real DE work.

PHASE 3 · SYSTEMS THINKING

Designing Data-Intensive Applications

Martin Kleppmann · O'Reilly 2017

Ch 1–4 and 10–12 in Phase 3. Dense but transformative. Changes how you think about reliability, scale, and tradeoffs.

PHASE 4 · PERFORMANCE

High Performance Python

Gorelick & Ozsvald · O'Reilly

Read selectively in Phase 4. Focus on profiling, memory management, and NumPy. Read it when you hit a performance problem, not before.

ONGOING · SUPPLEMENTAL

Kaggle Winning Solution Writeups

kaggle.com/discussions · Free

Read one top-3 solution writeup per week in Phase 4. Pay attention to how they describe their feature engineering decisions, not their models.
