🎓 A Career Mentor's Take on This Path

Nine months is enough to become job-ready for junior and entry-level DE roles — if you go deep rather than wide. The biggest mistake self-taught engineers make is trying to learn everything. The second biggest is finishing a course and thinking they're done. This roadmap does neither.

The four phases mirror how real DE careers develop: first you master the tools, then you build pipelines, then you move to the cloud, then you show your work and get hired. O'Reilly access changes your learning quality significantly — you're reading from the people who built these tools, not someone who summarised them. Use every page of it.

The skills you will have at the end of Phase 4 — Python, SQL, Airflow, dbt, Docker, BigQuery, CI/CD, data quality, and a production-grade capstone — are the exact skills that appear on the majority of junior-to-mid DE job descriptions globally, and specifically in Nigeria's growing tech sector.

Phase 01 · June 14 – August 1 · 7 Weeks
01

Foundations

Go slow here. Everything after this is built on top of this.

Jun 14 – Aug 1 Weeks 1–7 ~98 hrs Python · SQL · MySQL · PostgreSQL · pytest

Most beginners rush foundations. They watch tutorials at 2× speed and feel productive. That feeling is a trap. The goal of Phase 1 is to make Python and SQL feel like first languages — automatic, not looked up. Two databases from week one: PostgreSQL and MySQL are both required in professional DE roles. pytest is introduced here because testing is a professional discipline, not an afterthought you bolt on later. Seven weeks here is the right call — depth over speed.

WEEK 1 · Jun 14–20
Python for Data Work
File I/O, pandas fundamentals, list comprehensions, functions, error handling. Write code every single day. No passive watching.
WEEK 2 · Jun 21–27
Python Depth + MySQL
OOP basics, virtual environments, modules, JSON handling. Install MySQL alongside PostgreSQL. Learn dialect differences: backticks, LIMIT syntax, data types.
WEEK 3 · Jun 28–Jul 4
Python Consolidation
Reinforce lists, dicts, functions, and OOP. Build small scripts daily. Solidify what Weeks 1–2 introduced before moving to SQL.
WEEK 4 · Jul 5–11
SQL Foundations
SELECT, WHERE, JOIN, GROUP BY, ORDER BY across both MySQL and PostgreSQL. DBeaver setup. Write 20+ queries on real data. Know the basics cold.
WEEK 5 · Jul 12–18
SQL Mastery — Both Dialects
Window functions, CTEs, subqueries, multi-table JOINs, EXPLAIN plans. Write 40+ queries. Same query in both MySQL and PostgreSQL. Know where they differ.
WEEK 6 · Jul 19–25
Data Formats + Git
CSV vs JSON vs Parquet vs Avro — why each exists. Git workflow (branches, PRs, commit conventions). Start building your GitHub presence this week.
WEEK 7 · Jul 26–Aug 1
pytest + Project Build KEY
Write pytest unit tests for every function you've built. Aim for 80%+ coverage. Build P1-A and P1-B. Ship to GitHub with a CI badge.

Phase 1 Projects
P1-A · WEEK 7
Data Profiler CLI
Foundations
What You BuildA command-line tool that accepts any CSV file, PostgreSQL table, or MySQL table and returns a complete quality report: null percentages, inferred types, duplicate rows, value distributions, and a list of suspected data issues. Supports both databases via a --db-type flag.
Why It MattersData profiling is the first thing any DE does on a new dataset — in your first week at any real job, this will be useful. Building it yourself means you understand it, not just use it. You will reuse this tool in Phases 2 and 3.
TestingWrite pytest unit tests for every transform function. Aim for 80%+ code coverage. This is non-negotiable — it is how professionals write code.
StackPython (pandas, argparse, tabulate, sqlalchemy), PostgreSQL, MySQL, pytest, GitHub with a CI badge.
Run it on a dataset you have never seen before. It surfaces 3+ quality issues. Tests pass. A stranger can clone and run it in under 3 minutes.
P1-B · WEEK 7
SQL Analytics Engine
Foundations
What You BuildLoad a real multi-table dataset into both PostgreSQL and MySQL. Write 15 analytical queries covering aggregations, window functions (RANK, LAG, running totals), CTEs, JOINs, and conditional logic. Where the dialects differ, note it in a comment. Export as a clean HTML report with a business question per query.
Why It MattersSQL is the most-used skill in data engineering. Professional DE roles require fluency across multiple dialects. Writing the same query in both databases teaches you the differences — which is exactly what comes up in interviews at companies running mixed database stacks.
StackPostgreSQL, MySQL, DBeaver (free, connects to both), Python to render the HTML report. All SQL files committed to GitHub.
Every query runs correctly in both databases. Dialect differences are documented inline. The HTML report is readable by a non-technical stakeholder.
Phase 02 · August 2 – October 4 · 9 Weeks
02

Pipeline Engineering

The line between a script and a system is orchestration.

Aug 2 – Oct 4 Weeks 8–16 ~126 hrs Airflow · dbt · Docker · Data Modeling · CI/CD

Nine weeks for pipeline engineering because Airflow and dbt both deserve more depth than a single week each. Airflow takes a full day just to get running. dbt takes real project work to understand properly. The extra weeks go into depth on both, plus a full week on data modeling (which most tutorials skip) and a dedicated CI/CD week that makes your repos look like professional engineering, not hobby projects.

WEEK 8 · Aug 2–8
Apache Airflow — Setup & Basics
DAGs, tasks, operators, connections, scheduling, the Airflow UI. Install via pip, then reproduce in Docker. Expect a full day of setup pain — that is the learning.
WEEK 9 · Aug 9–15
Airflow — Depth
XComs, sensors, custom operators, task dependencies, backfills, retries, SLAs. Advanced patterns that appear in real pipelines. Build a multi-branch DAG.
WEEK 10 · Aug 16–22
Airflow Consolidation
Reinforce weeks 8–9 by building a complete real-world DAG from scratch. Debug and iterate. This is the week to break things and fix them.
WEEK 11 · Aug 23–29
Data Modeling
Star schemas, fact tables, dimension tables, slowly changing dimensions (SCD Types 1, 2, 3), normalisation tradeoffs. Design before you build — always.
WEEK 12 · Aug 30–Sep 5
dbt Core
Models, sources, refs, tests (not null, unique, accepted values), documentation, the lineage graph. Your first dbt project connected to your Phase 1 PostgreSQL data.
WEEK 13 · Sep 6–12
dbt — Depth
Macros, packages, incremental models, snapshots for SCD Type 2, dbt Cloud intro. dbt is how senior analytics engineers think. Go deep here — it pays dividends.
WEEK 14 · Sep 13–19
dbt Consolidation
Build a complete dbt project end-to-end: staging → intermediate → mart. Run dbt docs serve. Review every model. Fix gaps before moving to Docker.
WEEK 15 · Sep 20–26
Docker
Docker images, containers, docker-compose. Package your full pipeline stack — Airflow, PostgreSQL, dbt. Anyone clones the repo, runs one command, everything runs.
WEEK 16 · Sep 27–Oct 4
CI/CD KEY
GitHub Actions: run pytest on every PR, validate Airflow DAGs on push, run dbt tests on merge. This is how professional DE teams work. Make it green.

Phase 2 Projects
P2-A · WEEK 10
Automated Multi-Source Pipeline
Intermediate
What You BuildAn Airflow DAG on a daily schedule pulling from two public APIs and one CSV source. Merges the data, runs your P1-A profiler as a validation task, logs the quality report, and writes clean output to PostgreSQL. Task 3 only runs if Tasks 1 and 2 succeed. Retries configured on each task.
StackApache Airflow, PostgreSQL, Python (pandas, requests, sqlalchemy). Unit tests for all transform functions. Hosted on GitHub.
The DAG has run 5+ times automatically. Execution history visible in Airflow UI. You can trigger a manual backfill to a past date.
P2-B · WEEKS 13–14
dbt Transformation Layer
Intermediate
What You BuildA proper dbt project on top of P2-A's raw PostgreSQL data: staging models that clean raw tables, intermediate models that join and enrich, mart models as your star schema (1 fact table, 3+ dimension tables). dbt tests on every model. Generated documentation site with lineage graph. At least one incremental model and one snapshot.
Why It Mattersdbt is the transformation tool of choice on modern data teams. Knowing it well — not just the basics but macros, incremental models, and snapshots — is a genuine differentiator at the junior-to-mid DE level.
Stackdbt Core (free, pip install), PostgreSQL, GitHub. Run "dbt docs serve" to show the lineage graph.
"dbt docs serve" shows a full lineage graph. Every model has at least 2 tests. The star schema makes sense to someone who didn't build it.
P2-C · WEEK 15
Containerised Pipeline Stack
Intermediate
What You BuildPackage the entire Phase 2 system — Airflow, PostgreSQL, dbt — into a single docker-compose.yml. Anyone clones the repo, runs one command, and the full pipeline is running in under 5 minutes. Integration tests run inside Docker.
StackDocker Desktop, docker-compose, Dockerfile, GitHub with a clear README explaining the one-command startup.
Delete all local containers and data. Run "docker-compose up" from a clean state. Everything reproduces identically. Integration tests pass.
P2-D · WEEK 16 NEW
CI/CD Pipeline for Data Engineering
Intermediate
What You BuildThree GitHub Actions workflows on your Phase 2 repo: (1) On every PR — run pytest and block the merge if any test fails. (2) On push to main — validate all Airflow DAG files load without errors. (3) On merge to main — run the full dbt test suite and post a status comment to the PR.
Why It MattersCI/CD is a listed requirement on professional DE job descriptions. It also signals software engineering discipline — the market wants DEs who build like engineers, not like students. Most self-taught candidates do not have working CI/CD on any of their repos. You will.
StackGitHub Actions (free tier), pytest, YAML workflow files. No external services needed beyond GitHub itself.
Open a PR with a deliberately failing test. The CI run blocks the merge. Fix it. CI passes. The workflow is your proof of engineering practice.
Phase 03 · October 5 – December 27 · 12 Weeks
03

Cloud & Analytics Engineering

Move your thinking from your laptop to infrastructure that scales.

Oct 5 – Dec 27 Weeks 17–28 ~168 hrs GCP · BigQuery · MongoDB · AWS · Terraform · Data Lakes

Twelve weeks for the cloud phase because it covers the most ground. GCP and BigQuery are the primary cloud path — the free tier is the most generous for learning. MongoDB is added because NoSQL database ingestion appears explicitly in DE job descriptions and represents a genuinely different way of thinking about data. AWS gets a dedicated week because most companies run on it. Terraform closes out the cloud section because reproducible infrastructure is what separates a cloud user from a cloud engineer.

WEEK 17 · Oct 5–11
GCP Fundamentals
GCP console, IAM, Cloud Storage (GCS), Cloud SDK, service accounts. Free tier setup. Everything you build from Week 18 onwards lives here.
WEEK 18 · Oct 12–18
BigQuery
Columnar storage, partitioning, clustering, slot-based cost model, query optimisation, loading from GCS. Migrate your Phase 2 data here. Connect Looker Studio for free dashboards.
WEEK 19 · Oct 19–25
BigQuery Consolidation
Optimise queries, explore partitioning strategies, cost control. Get fully comfortable before moving to data lake design.
WEEK 20 · Oct 26–Nov 1
Data Lake Design
Data lake vs warehouse vs lakehouse. Medallion architecture: Bronze (raw) → Silver (clean) → Gold (business-ready Parquet). Design your lake on GCS and automate transitions with Airflow.
WEEK 21 · Nov 2–8
dbt + BigQuery — Analytics Eng.
Connect dbt to BigQuery. Rebuild your Phase 2 star schema on BigQuery. Add dbt exposures for your Looker Studio dashboard. This is full analytics engineering.
WEEK 22 · Nov 9–15
dbt + BigQuery Consolidation
Complete P3-A. Full lineage graph, tests on every model, Looker Studio connected. Terraform provisioning for all infrastructure.
WEEK 23 · Nov 16–22
MongoDB + NoSQL NEW
Document model vs relational. Collections, documents, indexes, aggregation pipeline. pymongo in Python. Change streams. When MongoDB, when PostgreSQL — understanding the tradeoff is the skill.
WEEK 24 · Nov 23–29
AWS Fundamentals NEW
S3, IAM, EC2 basics, RDS (managed MySQL + PostgreSQL on AWS), AWS CLI. Free tier throughout. Map every AWS concept to its GCP equivalent. Know both.
WEEK 25 · Nov 30–Dec 6
Terraform
Infrastructure as Code: provision GCS buckets, BigQuery datasets, and an S3 bucket using Terraform. Reproducible infrastructure — the same idea as Docker but for cloud resources.
WEEK 26 · Dec 7–13
Data Quality
Great Expectations: define expectation suites on your data lake layers, auto-generate data docs on every Airflow run, alert when quality checks fail.
WEEK 27 · Dec 14–20
Integration Week
Wire Phases 1–3 into one coherent architecture. Build P3-D: your full architecture document with diagram. Every system connected and documented.
WEEK 28 · Dec 21–27
Phase 3 Review + Rest
Review and polish all Phase 3 projects. Close any gaps. Make sure every repo has a README. Prepare for the capstone phase. Rest over the holidays.

Phase 3 Projects
P3-A · WEEKS 18 + 21–22
Cloud Analytics Platform
Advanced
What You BuildMigrate your Phase 2 Airflow pipeline to load data into BigQuery. Build your dbt project on BigQuery with a full star schema, tests, and lineage docs. Connect a Looker Studio dashboard showing 4+ meaningful business metrics. Partition the main fact table by date. Provision the entire infrastructure with Terraform — so it can be destroyed and reproduced from scratch in under 10 minutes.
Why It MattersBigQuery + dbt + Looker Studio is the most common modern analytics stack in the market. Knowing all three together, with Terraform underpinning it, is a genuine differentiator at the junior level.
StackGCP free tier, BigQuery, Cloud Storage, dbt (BigQuery adapter), Looker Studio (free), Terraform, GitHub Actions CI.

P3-B · WEEKS 20 + 26
Medallion Data Lake with Quality Monitoring
Advanced
What You BuildA three-layer data lake on GCS: Bronze (raw, exactly as ingested), Silver (cleaned, typed, validated), Gold (business-ready Parquet aggregates, partitioned by date). Airflow orchestrates the promotion between layers, running Great Expectations checks at each stage. If data quality fails at Silver, the Gold layer is not updated. Data docs auto-generate on every run.
StackGoogle Cloud Storage, Python (pandas, pyarrow), Apache Airflow, Great Expectations, Parquet, Terraform for bucket provisioning.
Introduce a deliberately bad record at the Bronze layer. Great Expectations catches it at Silver. The Airflow DAG fails gracefully. Gold data is untouched. Data docs show the failure.
P3-C · WEEK 23 NEW
MongoDB Ingestion Pipeline
Advanced
What You BuildA pipeline that reads documents from a MongoDB collection using pymongo, flattens the nested JSON structure, runs your P1-A profiler on the output, and writes results to both PostgreSQL and GCS as Parquet. An Airflow DAG schedules this daily. This simulates ingesting from a MongoDB-backed application database into your data lake — a pattern you will encounter at almost any product company.
Why It MattersMongoDB is explicitly listed in DE job descriptions. DEs regularly ingest from NoSQL application databases into relational warehouses. The pattern here — document store → flatten → relational warehouse — is used at fintech, SaaS, and e-commerce companies everywhere.
StackMongoDB Atlas (free tier — comes with sample datasets pre-loaded), pymongo, Python, PostgreSQL, GCS, Airflow, pytest.
Nested MongoDB documents are flattened and land correctly in both PostgreSQL and GCS Parquet. The Airflow DAG task graph succeeds. Tests pass on the flatten functions.
P3-D · WEEK 27
Integrated Architecture Document
Advanced
What You BuildA written architecture document (1–2 pages + diagram) that maps your entire Phase 1–3 system: data sources → ingestion → Bronze/Silver/Gold lake → BigQuery warehouse → dbt transformations → Looker Studio dashboard. Explain every design decision: why BigQuery over PostgreSQL for the warehouse, why Parquet in the lake, why dbt for transforms, why Airflow for orchestration. Include a "what I would do differently" section.
Why It MattersThe ability to document a system and defend design decisions is what senior DEs do in interviews. Writing this document forces clarity on everything you've built. It also becomes part of your capstone README in Phase 4.
FormatMarkdown file in your GitHub repo + a diagram using draw.io (free, no signup required). Link it from your GitHub profile README.
A non-technical person can read the document and understand what the system does and why it was built that way. A technical person can reproduce it from the document alone.
Phase 04 · December 28 – March 14 · 11 Weeks
04

Portfolio & Career Launch

Turn eight months of real work into a case you can make for yourself.

Dec 28 – Mar 14 Weeks 29–41 ~154 hrs Capstone · System Design · Resume · Open Source · Apply

Eleven weeks for the final phase. By now you have 11 projects, a working data lake, a cloud analytics platform, a MongoDB pipeline, CI/CD on every repo, and a written architecture document. Phase 4 is not a learning phase. It is a synthesis and presentation phase. The capstone takes 5 weeks because it deserves them. The remaining weeks are for system design preparation, resume work, open source contribution, and applications. You should be applying to roles in Week 39 at the latest — not after everything is "perfect".

WEEKS 29–33 · Dec 28–Jan 31
Production Capstone
Build P4-A. Your most important professional artefact. Five uninterrupted weeks. Pick a topic that genuinely excites you — that energy comes through in every conversation about it.
WEEK 34 · Feb 1–7
System Design Preparation
DE system design questions: design a pipeline for 1M events/day, batch vs real-time tradeoffs, data modeling under constraints, when to use which database. Practice out loud, not silently.
WEEK 35 · Feb 8–14
Resume + LinkedIn + Portfolio Polish
One-page resume, every project linked. LinkedIn updated with all GitHub links. GitHub profile README becomes your landing page. Make everything click-able and readable in under 60 seconds.
WEEKS 36–37 · Feb 15–28
System Design Practice + Open Source
Continue system design practice. One merged open source PR on Airflow, dbt, Great Expectations, or any tool from this roadmap. Start applying this week.
WEEKS 38–41 · Mar 1–14
Apply + Iterate
At least 10 applications: junior DE, analytics engineer, data analyst with pipeline responsibility, contract work on Upwork. Track everything. Iterate on your resume based on feedback.

Capstone
