The Intent

This is not a sprint. It is not a bootcamp, a certificate programme, or a performance for anyone else. It is six months of proving something to yourself — that you can go deep on a technical field, build real things, and come out the other side genuinely capable of a career in Data Engineering. The pace is human. The depth is real. The projects are the proof.

Foundations

Slow down here. Everything else is built on this.

June 1 – June 28Weeks 1–480–100 hrsPython · SQL · Git · Data Formats

Most people rush foundations. They watch tutorials at 2× speed and feel productive. Don't do that. The goal of this phase is to make Python and SQL feel like first languages — automatic, not looked-up. You're not preparing for a test. You're building intuition.

WEEK 1 · Jun 1–7
Python for Data Work
File I/O, pandas fundamentals, list comprehensions, functions, error handling. Write code every day.
WEEK 2 · Jun 8–14
Python — Depth
OOP basics, virtual environments, modules, JSON handling. Build two small utilities from scratch with no tutorial.
WEEK 3 · Jun 15–21
SQL Mastery
Window functions, CTEs, subqueries, multi-table JOINs, EXPLAIN plans. Write 40+ queries on real data.
WEEK 4 · Jun 22–28
Data Formats + Dev Workflow
CSV vs JSON vs Parquet vs Avro. Git, GitHub, CLI, project structure. Build P1-A and P1-B.

Phase 1 Projects
P1-A · WEEK 4
Data Profiler CLI
Foundations
What You BuildA command-line tool: point it at any CSV or database table and it returns a full quality report — nulls, type inference, duplicates, value distributions, suspected issues. You will reuse this in every phase that follows.
Why It MattersData profiling is the first thing a DE does on any new dataset. Building the tool yourself means you understand it, not just use it. It's a real portfolio artefact — not a tutorial copy.
StackPython (pandas, argparse, tabulate), PostgreSQL or SQLite, GitHub with a proper README.
Run it on a dataset you've never seen before. It surfaces at least 3 quality issues you didn't already know about.
P1-B · WEEK 4
SQL Analytics Engine
Foundations
What You BuildLoad a real multi-table dataset into PostgreSQL. Write 15 analytical queries — aggregations, window functions (RANK, LAG, running totals), CTEs, conditional logic. Export results as a clean HTML report with a business question per query.
Why It MattersSQL is the most-used skill in Data Engineering. Window functions and CTEs appear in almost every interview and every real pipeline. You need them to be automatic, not recalled.
StackPostgreSQL, DBeaver (free), Python to render the HTML report. All SQL files on GitHub.
Show someone the report without explaining the SQL. If they understand the insights, the queries are good.
Phase 02 · June 29 – July 26
02

Pipeline Architecture

The difference between a script and a system.

June 29 – July 26Weeks 5–880–100 hrsAirflow · dbt · Docker · Data Modeling

A hobbyist writes a script that runs when they run it. An engineer builds a system that runs whether they're watching or not. This phase is that shift. Expect to spend a full day just getting Airflow running — that struggle is part of the learning, not a sign something is wrong.

WEEK 5 · Jun 29–Jul 5
Apache Airflow
DAGs, tasks, operators, XComs, connections, scheduling, the Airflow UI. Install locally, then via Docker.
WEEK 6 · Jul 6–12
Data Modeling
Star schemas, fact vs dimension tables, slowly changing dimensions, normalisation tradeoffs. Design before you build.
WEEK 7 · Jul 13–19
dbt Core
Models, refs, sources, tests, documentation, lineage graphs. dbt is how modern data teams do SQL transformations.
WEEK 8 · Jul 20–26
Docker + Assembly
Images, containers, volumes, docker-compose. Containerise your Phase 1 tools. Wire everything into one system.

Phase 2 Projects
P2-A · WEEK 6
Automated Multi-Source Pipeline
Intermediate
What You BuildAn Airflow DAG on a daily schedule: pulls from two public APIs and one CSV source, merges the data, runs your P1-A profiler on the result, logs the quality report, and writes clean data to PostgreSQL. Task 3 only runs if Tasks 1 and 2 succeed.
Why It MattersThis is the core DE loop: ingest → validate → store → schedule. Every data engineering role involves some version of this. Airflow experience appears on nearly every JD.
StackApache Airflow, PostgreSQL, Python (pandas, requests). Your P1-A tool integrated as a validation step.
The DAG has run 3+ times automatically. You can see the execution history and trigger a manual backfill.
P2-B · WEEK 7
dbt Transformation Layer
Intermediate
What You BuildOn top of P2-A's raw PostgreSQL data, build a proper star schema using dbt: staging models, intermediate joins, final fact and dimension tables. Add dbt tests throughout and generate the documentation site with the lineage graph.
Why It Mattersdbt is how analytics engineers and data engineers collaborate on SQL transformations in a testable, version-controlled, documented way. It is now a standard tool on modern data teams.
Stackdbt Core (free, pip install), PostgreSQL, GitHub.
"dbt docs serve" shows a lineage graph a non-technical person can follow. All models have at least one test.
P2-C · WEEK 8
Containerised Pipeline Stack
Intermediate
What You BuildPackage your entire Phase 2 system — Airflow, PostgreSQL, dbt — into a single docker-compose.yml. Anyone should be able to clone your repo, run one command, and have the full pipeline running in under five minutes.
Why It MattersDocker is the line between a pipeline that works on your laptop and one you can actually ship. It comes up in every DE interview and every real team deployment.
StackDocker Desktop, docker-compose, Dockerfile, GitHub.
Delete your local database and containers. Run "docker-compose up" from scratch. Everything reproduces identically.
Phase 03 · July 27 – August 23
03

Cloud & Infrastructure

Move your thinking from your laptop to the internet.

July 27 – August 23Weeks 9–1280–100 hrsGCP · BigQuery · Data Lakes · Terraform

Every serious DE role touches cloud infrastructure. This phase is about developing a natural comfort with cloud platforms — not deep DevOps expertise, but enough to set up, use, and reason about cloud-native data storage and warehousing confidently. GCP's BigQuery free tier makes it the recommended starting point.

WEEK 9 · Jul 27–Aug 2
GCP Fundamentals
GCP console, IAM, Cloud Storage, Cloud SDK, service accounts. Free tier is generous. Set up once, use throughout.
WEEK 10 · Aug 3–9
BigQuery
Columnar storage, partitioning, clustering, cost model, query optimisation, loading from GCS. Your cloud warehouse.
WEEK 11 · Aug 10–16
Data Lakes
Data lake vs warehouse vs lakehouse. Medallion architecture (Bronze → Silver → Gold). Design before you build.
WEEK 12 · Aug 17–23
Terraform + Project
Infrastructure as Code basics. Terraform to provision GCS buckets and BigQuery datasets. Reproducible infrastructure.

Phase 3 Projects
P3-A · WEEK 10
Cloud Data Warehouse
Advanced
What You BuildMigrate your Phase 2 pipeline to load data into BigQuery. Build a Looker Studio dashboard (free, connects directly to BigQuery) with at least 4 meaningful charts. Partition the main table by date and show a partitioned query is faster and cheaper than a full table scan.
Why It MattersBigQuery is the most-used cloud data warehouse in the market. Partitioning and columnar storage prove you understand scale, not just syntax. Looker Studio means you can present to non-technical stakeholders — a DE skill as important as writing code.
StackGCP (free tier), BigQuery, Cloud Storage, Looker Studio, Python (google-cloud-bigquery SDK), Terraform for provisioning.
Your Airflow DAG writes to BigQuery. The Looker Studio dashboard refreshes automatically. You can explain why partitioning matters.
P3-B · WEEKS 11–12
Medallion Data Lake
Advanced
What You BuildDesign and build a three-layer data lake on GCS using the medallion architecture. Bronze: raw data exactly as ingested. Silver: cleaned, validated, typed. Gold: business-ready aggregates in Parquet. Automate the promotion between layers with Airflow.
Why It MattersThe medallion architecture is the dominant pattern for modern data lakes. Understanding why data moves through stages — and what is done in each — appears in almost every senior DE discussion and interview.
StackGoogle Cloud Storage, Python (pandas, pyarrow), Apache Airflow, Parquet, Terraform to provision the bucket structure.
You can draw the three layers on paper, explain what lives in each, and show an automated Airflow run promoting data from Bronze to Gold.
Phase 04 · August 24 – September 27
04

Scale & Advanced Engineering

What you do when the data is bigger than your RAM.

August 24 – September 27Weeks 13–17100–125 hrsPySpark · Kafka · Data Quality · Advanced Airflow + dbt

Phase 4 is where the engineering gets serious. Distributed processing, streaming data, data quality at scale, and the advanced patterns in Airflow and dbt that separate junior DEs from mid-level ones. Take your time here — these concepts reward patience more than speed.

WEEK 13 · Aug 24–30
PySpark Foundations
RDDs vs DataFrames, transformations vs actions, lazy evaluation, partitions. Run locally via Docker. Process a dataset too large for pandas.
WEEK 14 · Aug 31–Sep 6
PySpark — Applied
Joins, aggregations, UDFs, reading/writing Parquet. Spark SQL. Connect Spark to your GCS data lake from Phase 3.
WEEK 15 · Sep 7–13
Streaming with Kafka
Topics, producers, consumers, consumer groups, offsets. Set up via Docker. Build a producer → consumer → database pipeline.
WEEK 16 · Sep 14–20
Data Quality & Observability
Great Expectations — suites, validation, data docs. Automated alerts when pipelines produce bad data.
WEEK 17 · Sep 21–27
Advanced Airflow + dbt
Custom operators, sensors, TaskFlow API. dbt macros, packages, incremental models. Level up the Phase 2 work.

Phase 4 Projects
P4-A · WEEKS 13–14
Spark Processing Pipeline
Scale
What You BuildA PySpark job that reads a large dataset (1GB+) from your GCS data lake, runs aggregations and window functions, and writes results back to the Gold layer as optimised Parquet. Compare performance against an equivalent pandas script — measure the difference.
Why It MattersUnderstanding when and why to reach for Spark — instead of pandas — is a mark of DE maturity. The comparison exercise teaches the lesson better than any tutorial.
StackPySpark (Docker), Google Cloud Storage, Parquet, Python.
You can explain lazy evaluation, why Spark is slower on small data, and when you'd choose it over pandas. The numbers back up your explanation.
P4-B · WEEK 15
Streaming Data Pipeline
Scale
What You BuildA Kafka pipeline running in Docker: a Python producer emits events (simulated sensor readings or API events), a consumer reads them and writes to PostgreSQL, and a Streamlit page shows the last 100 events refreshing in real time.
Why It MattersStreaming is the fastest-growing area of DE. Even if your first role is batch-focused, understanding streaming vocabulary and architecture is expected at mid-level and above.
StackApache Kafka (Docker), Python (kafka-python), PostgreSQL, Streamlit.
Messages flow from producer to consumer to database without any manual steps. You can explain the difference between a Kafka topic and a database table.
P4-C · WEEKS 16–17
Data Quality Framework
Scale
What You BuildRetrofit Great Expectations into your Phase 2 or Phase 3 pipeline. Define expectation suites for each data source. Generate data docs that auto-update on every pipeline run. Add an Airflow task that fails the DAG and sends an alert when quality checks fail.
Why It MattersBad data is more expensive than no data. Data quality monitoring is one of the first things a professional DE team adds after a pipeline goes to production. Knowing Great Expectations is increasingly listed on mid-to-senior DE job descriptions.
StackGreat Expectations, Apache Airflow, PostgreSQL or BigQuery, Python.
Introduce a deliberately bad record into your data source. The pipeline catches it, fails gracefully, logs the issue, and leaves good data untouched.
Phase 05 · September 28 – October 25
05

Modern Stack & ML Interface

Learn how data engineering connects to artificial intelligence.

September 28 – October 25Weeks 18–2180–100 hrsSnowflake · Airbyte · Feature Eng · MLflow

By Phase 5 you have a real DE skillset. This phase adds two things: fluency with the modern commercial data stack that most companies actually use, and an understanding of how data engineers support machine learning teams. That second part is increasingly what separates DE generalists from DEs who can work anywhere in the AI economy.

WEEK 18 · Sep 28–Oct 4
Snowflake
Virtual warehouses, time travel, zero-copy cloning, Snowpipe, cost model. Free trial is sufficient. Understand why teams pay for it.
WEEK 19 · Oct 5–11
Modern Ingestion Tools
Airbyte (open-source EL tool). The ELT pattern — why modern teams extract and load first, transform later with dbt.
WEEK 20 · Oct 12–18
Feature Engineering for ML
What ML models actually need from data. Feature stores (Feast). Train/test splits without leakage. Parquet output pipelines.
WEEK 21 · Oct 19–25
MLflow + ML Pipelines
Experiment tracking, model registry, artifact storage. Build a pipeline that feeds a simple model and logs the run. Manage the data, not the model.

Phase 5 Projects
P5-A · WEEKS 20–21
Feature Engineering Pipeline
Advanced
What You BuildTake a tabular dataset with a clear prediction task. Build a full Python pipeline: ingest → profile (P1-A) → clean → engineer 10+ meaningful features with a written justification for each → output train.parquet and test.parquet. Log the feature creation run in MLflow. The Parquet files should be the only thing an ML engineer needs from you.
Why It MattersThis is the highest-value intersection between DE and ML. Most ML teams spend more time on data than on models. A DE who understands this handoff — and can execute it cleanly — is genuinely rare and genuinely sought after.
StackPython (pandas, scikit-learn, pyarrow), MLflow, Parquet, GitHub with a full README explaining every feature decision.
An ML engineer could clone your repo, run one script, and get clean Parquet files. The README explains every feature without assuming DE knowledge.
P5-B · WEEK 21 · STRETCH
End-to-End ML Data Pipeline
Stretch Goal
What You BuildExtend P5-A into a scheduled Airflow pipeline: raw data lands in GCS → Bronze → Silver → Gold → feature engineering → Parquet output → MLflow logs the run. A complete automated path from raw data to model-ready files, with quality checks at every stage.
StackEverything from Phases 1–5. This is a synthesis project, not a new technology project.
The pipeline runs on schedule without intervention. MLflow shows a new run entry each time. Parquet files are dated and versioned.

Phase 6 Projects
P6-A · WEEKS 22–24
Production Capstone System
Capstone
What You BuildA complete, production-grade data engineering system on a topic that genuinely interests you. It must include: automated ingestion from at least two sources, a cloud data warehouse (BigQuery), a medallion data lake on GCS, dbt transformations with tests, data quality monitoring with Great Expectations, an Airflow schedule, and a public dashboard. Everything runs in Docker and lives on GitHub with a README that reads like a real project.
Why This Is the Most Important Thing You Will BuildThis is what you send instead of a CV. Not certificates. Not a DataCamp profile. This project. One strong, real, end-to-end system speaks more clearly than a list of tools on a resume. Pick a topic you genuinely care about — that comes through when you talk about it.
Topic IdeasNigerian energy grid vs weather correlation · African fintech transaction patterns · Agricultural yield and market data for West Africa · NPFL (Nigerian Premier League) football analytics · Public health indicator trends across sub-Saharan Africa · Anything that gives you energy when you think about it.
StackPython, Airflow, dbt, BigQuery, GCS, Great Expectations, Docker, Looker Studio or Metabase, Terraform, GitHub.
You can walk through this project in a 5-minute live demo to people who don't know you. The README sells it before you open your mouth. The repo is public.
P6-B · WEEK 26
Open Source Contribution
Career
What You BuildOne meaningful contribution to an open source DE project — Apache Airflow, dbt, Great Expectations, or any tool used in this roadmap. A bug fix, improved documentation, or a new example all count. Open a pull request and see it through to merge.
Why It MattersAn open source contribution signals that you understand collaborative software development — PRs, reading others' codebases, communicating with maintainers. It's rare enough among applicants to be genuinely noticed.
A merged pull request exists on a project you didn't create. Link it on your resume and LinkedIn.
