The Blueprint

This roadmap bridges the gap between learning syntax and deploying production-grade data infrastructure. It heavily indexes on Python, SQL, CI/CD, Airflow orchestration, dbt modeling, and cloud data warehousing. Timeline runs from mid-June to early October, accommodating a healthy 20–25 hour weekly rhythm.

📚 Core Offline Library & Resource Access

Since you have offline access to the O'Reilly archives and an active DataCamp scholarship, the prior time constraint is lifted. Proceed deliberately. Map your reading of Fundamentals of Data Engineering and Designing Data-Intensive Applications directly to the respective pipeline and infrastructure phases below.

Foundations

Syntax must become automatic. Build tools, not just scripts.

June 15 – July 12 Weeks 1–4 Python · SQL · MySQL · PostgreSQL + pytest
WEEK 1 · Jun 15–21
Python for Data Work
  • Control flow, functions, error handling (Try/Except logic).
  • File I/O (CSV and JSON mapping).
  • Pandas fundamentals: Series, DataFrames, filtering, group by.
Milestone: Write a script to clean a messy flight manifest CSV and output a summarized JSON without hardcoding paths.
WEEK 2 · Jun 22–28
Python Depth + MySQL
  • OOP basics: classes, methods, inheritance for data objects.
  • Virtual environments (venv/Conda) and dependency isolation.
  • Install MySQL locally alongside PostgreSQL.
Milestone: Connect Python to MySQL using sqlalchemy and execute basic CRUD operations on customer records.
WEEK 3 · Jun 29–Jul 5
SQL Mastery (Both DBs)
  • Window functions: RANK, DENSE_RANK, LAG, LEAD.
  • Common Table Expressions (CTEs) for logic segmentation.
  • Complex multi-table JOINs and reading EXPLAIN plans.
Milestone: Solve 10 complex analytical questions on a multi-table schema in both PostgreSQL and MySQL dialects.
WEEK 4 · Jul 6–12
Data Formats + Git + pytest
  • CSV vs JSON vs Parquet memory/speed profiling.
  • Git workflow: branching, staging, committing via CLI.
  • Writing assertions and mocking data logic with pytest.
Milestone: Push your code to GitHub with a passing pytest suite for your cleaning functions.
Phase 1 Projects
P1-A · WEEK 4
Data Profiler CLI
Foundations
ObjectiveBuild a command-line utility that points at any CSV, PostgreSQL, or MySQL table and returns a full statistical quality report in the terminal.
ArchitecturePython using argparse for CLI inputs, pandas for data inference, and tabulate to render the output table. Must support a --db-type flag to switch database connections.
The Hard PartDynamically handling messy data types (like mixed strings and ints in a single column) without Pandas throwing a runtime exception.
Success Criteria80%+ pytest coverage on the core logic. Run pytest tests/ and see all green before pushing to Git.
P1-B · WEEK 4
SQL Analytics Engine
Foundations
ObjectiveProve multi-dialect SQL fluency. Load a real multi-table dataset into both DBs and write complex business logic queries.
Airline ContextUse an open-source flight delay or ticketing dataset. Write queries to find "Top 5 routes with highest consecutive delays (using LAG/LEAD)" or "Customer lifetime value tiers (using CTEs)."
The Hard PartTranslating date/time string manipulation functions between PostgreSQL syntax and MySQL syntax.
Success CriteriaExport results as an HTML report generated by Python. Both database queries yield the exact same numerical outputs.

Phase 2 Projects
P2-A · WEEK 6
Automated Multi-Source Pipeline
Intermediate
ObjectiveBuild a daily Airflow DAG that pulls from a REST API and a static CSV, merges them, runs your P1-A profiler as validation, and loads to PostgreSQL.
ArchitectureTask 1 (Extract API) → Task 2 (Extract CSV) → Task 3 (Validate) → Task 4 (Load DB). Task 4 only runs if all preceding tasks succeed.
The Hard PartMaking the DAG idempotent. If you run the DAG three times for the same date, you should not get triplicate records in your database.
P2-B · WEEK 6
dbt Transformation Layer
Intermediate
ObjectiveTransform the raw tables created in P2-A into a business-ready Star Schema using dbt.
ArchitectureBase staging layer → Intermediate joins → Final Fact/Dimension models. YAML files configured to run not_null and unique tests.
The Hard PartUnderstanding how dbt's ref() function completely replaces hardcoded DDL/DML statements, shifting your brain into a declarative mindset.
P2-C · WEEK 7
Containerised Pipeline Stack
Intermediate
ObjectiveEncapsulate your entire workflow so another engineer can run it on their machine with zero dependency hell.
ArchitectureA docker-compose.yml file orchestrating the Airflow webserver, scheduler, and the PostgreSQL database, all communicating over a custom Docker network.
The Hard PartNetworking. Ensuring Airflow inside Container A can successfully talk to PostgreSQL running in Container B via internal hostnames.
P2-D · WEEK 8
CI/CD Pipeline for Data Engineering
Intermediate
ObjectiveImplement professional software engineering checks on your codebase.
ArchitectureThree GitHub Actions workflows: (1) On PR: run pytest unit tests. (2) On Main Push: validate DAG import errors. (3) On Merge: run the dbt test suite.
The Hard PartSetting up the GitHub Actions runner environment to accurately mimic your local Docker environment so tests don't falsely fail.

Phase 3 Projects
P3-A · WEEK 10
Cloud Data Warehouse
Advanced
ObjectiveMigrate your local PostgreSQL workflow to a cloud-native, scalable warehouse using Google BigQuery.
ArchitecturePython scripts interacting with the google-cloud-bigquery SDK to load data. Use Looker Studio to visualize the final tables directly from BigQuery.
The Hard PartAdapting your DDL statements to BigQuery's specific dialect and enforcing partition schemas during data insertion.
P3-B · WEEKS 11–13
Medallion Data Lake Infrastructure
Advanced
ObjectiveTreat infrastructure as code. Provision the storage buckets for a data lake algorithmically, not by clicking around a UI.
ArchitectureTerraform files (.tf) defining three distinct GCS buckets (Bronze, Silver, Gold) with specific IAM permissions and lifecycle deletion rules.
The Hard PartManaging the Terraform state file locally without accidentally destroying active cloud resources during testing.
P3-C · WEEK 12
MongoDB Application Ingestion
Advanced
ObjectiveReplicate the reality of modern startups: pulling operational data out of a NoSQL application database into an analytical environment.
ArchitectureAirflow DAG utilizing pymongo to query a MongoDB Atlas cluster, extracting nested JSON (e.g., flight booking metadata + array of passenger details), flattening it with Python, and loading it to BigQuery.
The Hard PartWriting efficient recursive logic to unnest dictionaries within lists without blowing up memory.
